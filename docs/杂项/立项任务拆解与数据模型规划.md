# 检查报告语言类质控项目：任务拆解与数据模型规划

## 1. 立项初衷转译为明确目标

根据谈话记录，项目的核心目标可归纳为以下 4 条：

1. 对检查报告进行语言类质控，覆盖三段文本：描述、检查所见、检查提示。
2. 质控链路必须先于病历生成，只有通过或修正后的数据才能进入下游生成。
3. 同时覆盖两类报告：指标类、影像类，并处理缺失项与矛盾项。
4. 建立可度量的效果对比体系：基座模型对比、是否人工参与、是否引入质控。

一句话定义：先把报告判对错并可解释，再生成病历。

---

## 2. 需要完成的任务清单

## 2.1 任务 A：报告标准化与输入校验

目标：保证上游输入可被稳定处理。

子任务：
1. 定义并校验统一输入结构（report_id、report_type、report_subtype、content、label）。
2. 对内容字段描述、检查所见、检查提示做缺失补齐与空值归一化。
3. 输出可追溯的校验日志（字段缺失、类型错误、非法报告类型）。

验收：
1. 非法样本不会进入后续模型链路。
2. 所有合法样本能转为统一内部格式。

## 2.2 任务 B：实体识别与关系抽取（NER/RE）

目标：把自然语言报告转为可计算结构。

子任务：
1. 指标类抽取：指标名、数值、单位、状态（正常/异常）。
2. 影像类抽取：病变实体、解剖部位实体、病变-部位关系（如 located_at）。
3. 抽取结果必须返回降级标记（degraded）与来源（source），便于稳定性监控。

验收：
1. 指标类抽取结果可直接用于阈值规则判定。
2. 影像类抽取结果可直接用于病变-解剖一致性校验。

## 2.3 任务 C：规则质控（主动干预）

目标：构建规则库并作为主导质控引擎。

子任务：
1. 完整性质控：检查描述/检查所见/检查提示缺失。
2. 指标逻辑质控：数值是否越界，结论与数值是否冲突。
3. 影像逻辑质控：所见与提示是否冲突，病变与解剖是否匹配。
4. 规则输出统一 issue 结构（type、severity、message、evidence、source）。

验收：
1. 每条问题都可定位证据。
2. 规则库可配置化维护，不写死在代码里。

## 2.4 任务 D：模型意见层（LLM reasoning）

目标：在规则质控后，让模型给出补充意见而非替代规则。

子任务：
1. 将报告文本、抽取结果、规则问题共同输入模型。
2. 输出判定结果（pass/fail/unknown）与理由。
3. 推理失败时不阻断流程，保留 unknown 与失败原因。

验收：
1. 推理层对规则结果提供增益（召回提升或误报下降）。
2. 异常场景下链路可降级运行。

## 2.5 任务 E：报告分级与病历生成

目标：形成先质控后生成的闭环。

子任务：
1. 按优先级分级：矛盾级 > 缺失级 > 正常级。
2. 仅允许正常级或已修正数据进入病历生成。
3. 保留原文、抽取、质控、推理、分级、生成结果的全链路记录。

验收：
1. 病历生成输入集可追溯到质控结果。
2. 被拦截样本可输出修复建议。

## 2.6 任务 F：评测与对比实验

目标：回答项目价值问题：质控是否带来稳定收益。

必做对比维度：
1. 基座模型对比：不同基础模型在抽取与推理任务上的效果。
2. 有无人工：纯自动流程 vs 人工复核介入流程。
3. 有无质控：直接生成 vs 质控后生成。

验收：
1. 对比报告可量化展示准确性与风险差异。
2. 每个维度至少包含一组可复现实验配置。

## 3. 需要什么样的数据集

项目至少需要 4 层数据集，建议分开构建、联合评估。

## 3.1 原始语料层（Raw Reports）

用途：训练/评测抽取与质控前处理。

建议字段：
1. report_id
2. report_type（指标类/影像类）
3. report_subtype（如血常规、胸部CT）
4. content.描述
5. content.检查所见
6. content.检查提示
7. 来源信息（科室、设备、时间，可脱敏）

质量要求：
1. 真实分布覆盖多科室与多写作风格。
2. 包含短文本、模板化文本、噪声文本。

## 3.2 信息抽取标注层（NER/RE Labels）

用途：训练或评估实体识别与关系抽取模型。

指标类建议标注：
1. 实体：indicator_name、value、unit、status。
2. 关系：indicator_name-value、value-unit、indicator_name-status。

影像类建议标注：
1. 实体：lesion、anatomy、modifier（可选）。
2. 关系：lesion-located_at-anatomy、lesion-has_modifier-modifier（可选）。

标注格式建议：
1. 文本级 JSON（便于与现有工程对接）。
2. 同时保留 span 起止位置（便于训练序列标注模型）。

## 3.3 质控标注层（QC Issues）

用途：训练质控分类器或评估规则+模型联合效果。

建议字段：
1. issue_type：缺失/矛盾/正常。
2. issue_subtype：字段缺失、数值结论冲突、所见提示冲突、病变部位不匹配等。
3. severity：high/medium/low。
4. evidence：触发文本片段、触发实体、触发规则ID。
5. final_judgement：是否确认为有效问题。

关键要求：
1. 每个问题必须有证据。
2. 支持多问题并存（同一报告可同时缺失+矛盾）。

## 3.4 生成评测层（Case Generation Eval）

用途：评估病历生成质量与质控收益。

建议字段：
1. 输入版本：原始输入/质控过滤后输入。
2. 输出病历文本。
3. 人工评分：事实一致性、完整性、可读性、安全性。
4. 自动指标：关键事实命中率、冲突残留率。

## 4. 训练什么样的模型

结论：本项目应采用规则主导 + 模型增强的分层策略，而不是单一端到端模型。

## 4.1 必备模型类型

1. 抽取模型（按类型分流）
- 指标类：结构化抽取模型，重点保障数值与单位准确率。
- 影像类：医学关系抽取模型，重点保障病变-解剖关系准确率。

2. 质控判别模型（可选增强）
- 输入报告文本 + 抽取结构 + 规则命中项。
- 输出问题类型与置信度。
- 用于补充规则盲区，不替代规则。

3. 推理意见模型
- 输出可解释结论（pass/fail/unknown）与文字理由。
- 主要承担复杂语义冲突判断和解释生成。

4. 病历生成模型
- 仅接收通过质控或已修正的数据。
- 强约束生成，避免引入未证实事实。

## 4.2 训练与优化顺序

推荐顺序：
1. 先做规则库与抽取质量基线。
2. 再训练/微调抽取模型，提高结构化准确率。
3. 再引入质控判别模型，优化规则盲区。
4. 最后优化病历生成模型，并基于质控输出做输入约束。

原因：若抽取与质控不稳定，病历生成会放大错误。

## 4.3 训练数据规模建议（初期）

1. 抽取标注集：每类报告 2000 到 5000 条高质量标注。
2. 质控标注集：至少 3000 条，且缺失/矛盾/正常分布尽量均衡。
3. 生成评测集：500 到 1000 条人工双评样本。

说明：早期可先小规模高质量数据起步，优于大规模弱标注。

---

## 5. 关键评估指标

## 5.1 抽取层

1. 实体级 F1（按实体类型拆分）。
2. 关系级 F1（重点看 lesion-located_at）。
3. 数值准确率（指标值与单位联合准确率）。

## 5.2 质控层

1. 缺失识别 Precision/Recall/F1。
2. 矛盾识别 Precision/Recall/F1。
3. 高风险问题漏检率（必须重点控制）。

## 5.3 生成层

1. 事实一致性得分。
2. 冲突残留率（越低越好）。
3. 医师可用性评分。

## 5.4 业务层

1. 质控拦截率。
2. 错误病历下发率。
3. 人工复核工时变化。

---

## 6. 对比实验设计（立项必答题）

建议固定同一批测试集，做以下四组：

1. 基座模型A，无人工，无质控。
2. 基座模型A，无人工，有质控。
3. 基座模型A，有人工，有质控。
4. 基座模型B，无人工，有质控。

可回答的问题：
1. 质控本身的增益（2 对比 1）。
2. 人工参与的边际价值（3 对比 2）。
3. 基座模型差异在质控框架下是否仍显著（4 对比 2）。

## 7. 与当前仓库实现的映射

已具备能力：
1. 指标类与影像类分流抽取。
2. 规则质控与 LLM 推理质控串联。
3. 分级输出（正常级/缺失级/矛盾级）。
4. 病历与体检总结生成。

建议补齐的关键项：
1. 增加规则版本管理与规则命中统计看板。
2. 增加标准化标注数据集（NER/RE/QC）与标注规范文档。
3. 增加系统化对比实验脚本（有无质控、有无人工、不同基座模型）。
4. 增加质控后再生成的硬门控配置（禁止问题样本直接生成）。

## 8. 里程碑建议（8 周样例）

1. 第 1-2 周：规则库v1 + 数据规范 + 基线评测集。
2. 第 3-4 周：NER/RE 标注与抽取模型优化。
3. 第 5-6 周：质控模型增强与错误分析闭环。
4. 第 7 周：病历生成门控上线与灰度验证。
5. 第 8 周：完成对比实验报告与立项验收材料。
